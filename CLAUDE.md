# 阶段路由器 + 强制输出模板

**每次收到消息，按以下步骤执行：**

1. 读取 `agent-doc/workflow-status.md`，找到 `当前阶段`
2. 找到下面与该阶段对应的模板
3. **在聊天窗口输出该模板的完整内容**（填充所有字段），不得添加模板外的内容
4. **将模板完整内容同时保存到对应文件**（路径见下方"文件保存强制规则"表），聊天窗口和文件两者缺一不可
5. 输出完成后，更新 `agent-doc/workflow-status.md` 中的阶段和下一步
6. **每次回复结束后，自动执行 Git 提交**（规则见下方）

> **所有模板均为强制。发现字段留空或用"..."、"略"、"同上"等省略词，视为违规。**

### 文件保存强制规则（所有阶段适用）

**每个模板填充完毕后，必须将模板内容同时保存到对应的文件中，不得仅输出在聊天窗口。**

| 计划拆分 | 架构设计文档 | `agent-doc/architecture/{yyyy-MM-dd}-{seq}-{项目}-architecture.md` |
| 计划拆分 | 技术设计文档 | `agent-doc/technical-design/{yyyy-MM-dd}-{seq}-{项目}-technical-design.md` |

### Git 自动提交规则（所有阶段适用）

**每次回复结束时自动执行 Git 提交，不得遗漏。**

| 规则项 | 说明 |
|--------|------|
| 触发条件 | 每次回复结束后（模板输出 + 文件保存 + workflow-status.md 更新全部完成后） |
| 执行操作 | `git add . && git commit -m "类型(范围): 描述"` |
| 提交信息格式 | 遵循 Conventional Commits：`类型(范围): 描述` |
| 提交时机 | 当前回复的所有产出物（文件 + 状态更新）完成后立即执行 |
| 推送策略 | 如有配置远程仓库，commit 后执行 `git push`；否则仅 commit |
| 注意事项 | 确保 .gitignore 已配置，不提交敏感文件或构建产物 |

> **例外**：纯对话交互（仅确认/问答，无任何文件变更）可不触发提交。

### 提示词工程师使用规则
**除以下明确简短指令外，所有用户请求必须先走"提示词工程师"模板进行请求增强**：

- `确认` / `可以` / `确认执行` / `停止执行`
- 其他语义明确的简短确认/否定指令（如 `是`、`否`、`继续`、`下一步`）

> 对于任何实质性需求描述（修改、添加功能、修复等），必须先输出"提示词工程师 Agent — 请求增强"模板进行分析，不得跳过。**如同时有实质性需求 + 简短确认，仅简短确认部分可直接执行，实质性需求仍需走提示词增强。**

### 自动化闭环流程概览
整个工作流中**仅以下3处需用户确认**，其余环节全自动闭环执行：

| 序号 | 确认节点 | 说明 |
|------|---------|------|
| 1 | 提示词增强后 | 用户确认意图分析是否准确 |
| 2 | 结果蓝图输出后 | 用户确认4个维度的终态设计是否完整 |
| 3 | 子计划拆分清单输出后 | 用户确认子计划是否符合蓝图要求 |

**3处确认之后，子计划执行阶段由核查Agent和稽查Agent自动把关：**
- 通过 → 自动继续下一子计划
- 不通过 → 自动通知PM Agent修正 → 修正后重新走完整核查+稽查流程 → 通过后继续
- 全部子计划执行完毕后 → 核查Agent重新读取终态蓝图文档，逐项检查蓝图所有详细内容是否被子计划涵盖
- 有遗漏 → 通知PM Agent安排修复 → 修复后重复相同检查，直到无遗漏
- 无遗漏 → 通知用户进行终态检查
