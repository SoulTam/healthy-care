## 1. 架构风格选择

| 决策 | 选择 | 理由 |
|------|------|------|
| 架构风格 | 分层单体 → 渐进式模块化 | 开发阶段单机部署，FastAPI 单体起步；按领域分包（user/constitution/diet/content/community），为未来拆微服务预留边界 |
| 前后端分离 | RESTful API | 前端 App/小程序共用同一套 API，BFF 模式暂不需要（单客户端类型），后续视复杂度引入 |
| API 版本策略 | URL路径版 `v1` | 简单直接，向后兼容通过新增字段而非改接口 |

## 3. 模块划分与职责

### 3.1 应用层模块

| 模块 | 目录 | 核心职责 | 对外依赖 |
|------|------|---------|---------|
| 用户模块 | `app/user/` | 注册/登录/鉴权/用户信息/饮食禁忌 | 业务DB |
| 体质评估模块 | `app/constitution/` | 问卷管理/评估计算/对话问诊/报告生成 | 业务DB + LLM服务 |
| 食补方案模块 | `app/diet/` | 方案生成/替换/查询/食谱管理 | 业务DB + 检索服务 |
| 内容模块 | `app/content/` | 养生知识/食材百科/科普卡片 | 业务DB |
| 反馈模块 | `app/feedback/` | 反馈提交/分析/闭环 | 业务DB + LLM服务 |
| 社区模块 | `app/community/` | 帖子/评论/点赞/审核（二期） | 业务DB + LLM服务 |
| 通知模块 | `app/notification/` | 通知管理/推送/设置 | 业务DB |
| 收藏模块 | `app/favorite/` | 收藏/取消/列表 | 业务DB |
| 搜索模块 | `app/search/` | 全局搜索/热词 | 业务DB + 检索服务 |
| 家庭模块 | `app/family/` | 成员管理/家庭方案（二期） | 业务DB + 检索服务 |
| 管理后台 | `app/admin/` | 内容CRUD/审核/统计/日志 | 业务DB + 检索服务 |

### 3.2 AI 引擎层模块

| 模块 | 目录 | 核心职责 |
|------|------|---------|
| 检索服务 | `engine/retrieval/` | 意图解析/条件构建/元数据过滤/语义检索/排序组合 |
| 嵌入服务 | `engine/embedding/` | 文本向量化/批量入库向量化 |
| LLM 服务 | `engine/llm/` | 统一LLM调用封装/流式响应/提示词模板管理 |

### 3.3 模块间依赖规则

```
应用层模块 → LLM服务:  仅通过 engine/llm/ 调用，禁止直接请求 Ollama API
应用层模块 → 检索服务: 仅通过 engine/retrieval/ 调用，禁止直接操作向量库
应用层模块 → 嵌入服务: 仅通过 engine/embedding/ 调用
AI 引擎层内部: LLM服务 ← 检索服务（意图理解）/ 嵌入服务 ← 检索服务（入库向量化）
```

> 参考：interface-selection-strategy.md — 同一进程内领域边界明确的模块通过接口/函数调用解耦，不跨进程则不用 HTTP API。

## 4. 数据架构

### 4.1 数据分类与存储

| 数据类型 | 存储位置 | 格式 | 说明 |
|---------|---------|------|------|
| 用户数据 | 业务DB | 关系表 | users / assessments / plans / feedback |
| 食谱元数据 | 业务DB | 关系表+JSON字段 | recipes / recipe_tags / recipe_ingredients |
| 知识切片 | 业务DB + 向量DB | 关系表 + 向量 | knowledge_chunks + Chroma 向量 |
| 向量数据 | Chroma / Milvus | 向量+metadata | 仅存储 embedding + ID + 基础标签 |
| 缓存数据 | Redis（生产） | KV | 热食谱缓存/会话缓存 |
| 离线数据 | 客户端本地 | AsyncStorage / SQLite | 已生成方案/已浏览百科 |

### 4.2 数据一致性策略

| 场景 | 策略 |
|------|------|
| 用户注册→创建初始记录 | 同步事务（PostgreSQL 单库事务） |
| 方案生成→写入 plan + meals | 同步写（单事务） |
| 反馈提交→更新权重 | 同步写反馈 + 异步更新偏好权重 |
| 资料入库→业务DB + 向量DB | 先写业务DB → 成功后写向量DB（最终一致） |
| 社区帖子→AI审核 | 异步：发帖后立即返回 → 后台审核 → 更新状态 |

## 5. 安全架构

| 层级 | 措施 | 说明 |
|------|------|------|
| 传输 | HTTPS (TLS 1.3) | Nginx 层终止 |
| 鉴权 | JWT (access + refresh token) | access 24h / refresh 30d |
| 密码 | bcrypt 哈希存储 | 不存明文 |
| 健康数据 | AES-256 加密 | 体质/症状/反馈敏感字段 |
| API 防护 | 限流 (Rate Limiting) | 每用户 60 req/min |
| 输入校验 | Pydantic model 校验 | 所有请求体严格校验 |
| 内容安全 | 敏感词双层过滤 | 客户端预过滤 + 服务端语义过滤（社区） |

## 6. 部署架构

### 6.1 开发环境（本机）

```
[单机]
FastAPI (uvicorn, port 8000)
PostgreSQL (port 5432)
Chroma (port 8001)
Ollama (port 11434)
BGE-small-zh (CPU, 本地加载)
```

### 6.2 生产环境（云服务器）

```
[Nginx] (port 443)
    │
    ├── FastAPI x 2 (Gunicorn + uvicorn workers)
    ├── PostgreSQL (port 5432)
    ├── Redis (port 6379)
    ├── Chroma/Milvus (port 19530)
    ├── vLLM (port 8000, GPU, Qwen2.5-7B-AWQ-INT4)
    └── BGE-base-zh (GPU/CPU)
```

### 6.3 配置管理

> 参考：cloud-design-patterns.md - External Configuration Store

配置通过环境变量 + `.env` 文件管理，不硬编码在代码中：

| 配置项 | 环境变量 | 开发默认值 |
|--------|---------|-----------|
| 数据库URL | `DATABASE_URL` | `postgresql+asyncpg://healthy_care:healthy_care@localhost:5432/healthy_care` |
| 向量库路径 | `CHROMA_PERSIST_DIR` | `./data/chroma` |
| Ollama URL | `LLM_BASE_URL` | `http://localhost:11434` |
| LLM 模型 | `LLM_MODEL` | `qwen2.5:7b` |
| Embedding 模型 | `EMBEDDING_MODEL` | `BAAI/bge-small-zh-v1.5` |
| JWT Secret | `JWT_SECRET` | （从环境变量读取，不设默认值） |
| 加密密钥 | `ENCRYPTION_KEY` | （从环境变量读取，不设默认值） |

## 7. 关键设计决策记录

| 编号 | 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|------|
| ADR-001 | 单体vs微服务 | 单体/微服务 | 分层单体 | 开发阶段单机部署，8核16线程32G内存，单体够用；模块分包预留拆微服务边界 |
| ADR-002 | BFF 是否需要 | 需要/不需要 | 不需要 | 当前仅 App+小程序，React Native 和微信小程序差异可通过各自端适配层处理；后续客户端类型增多（Web）再引入 |
| ADR-003 | LLM 推理框架 | Ollama/vLLM | 开发Ollama，生产vLLM | Ollama 本地部署简单，vLLM 生产吞吐量高 |
| ADR-004 | 向量数据库 | Chroma/Milvus/Pinecone | 开发Chroma，生产Milvus | Chroma 零配置适合开发，Milvus 生产级性能 |
| ADR-005 | 业务数据库 | SQLite/PostgreSQL/MySQL | 开发/生产统一 PostgreSQL | 与生产方言、JSON、UUID、索引能力保持一致，降低环境差异风险 |
| ADR-006 | 前端跨平台 | React Native/Flutter/原生 | React Native | 一套代码 iOS+Android，社区成熟，生态丰富 |
| ADR-007 | Embedding 模型选型 | BGE-small-zh / BGE-base-zh / BGE-large-zh | 开发small，生产base | small CPU 可运行且足够（384维），base 精度更高（768维） |
| ADR-008 | LLM 仅做 NLU+格式化 | 是/否 | 是 | 核心原则：不做超越资料的推理，LLM 不参与食补知识生成 |
| ADR-009 | 缓存策略 | Redis/内存/无 | 生产Redis，开发无 | 开发阶段数据量小无需缓存，生产用 Redis Cache-Aside 模式 |
| ADR-010 | 离线策略 | 本地缓存全部/仅缓存已生成 | 仅缓存已生成+百科 | 尊重数据边界，不预缓存全部数据 |
